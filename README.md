# Visual Analytics for Atomic Fact Annotation

Prototype implementation for the MSc thesis
*"Visual Analytics for Atomic Fact Annotation: Guideline Refinement through
a Conflict-Aware Knowledge Graph"* (Zilong Yan, University of Konstanz,
DBvis group).

This README is meant to get a fresh checkout running on someone else's
machine in ~15 minutes. For thesis context, scope, and design decisions,
read `PROJECT_STATE.md` and `doc/thesis_proposal.tex` instead.

---

## 1. What's in the repo

```
src/                LLM extractor, alignment, conflict layers, KG builder, parsers
scripts/            CLI batch drivers (dry-run, extraction, phase-2, doc selection)
ui/                 Flask + Cytoscape.js single-page visual-analytics UI
prompts/            extract_v1.md / extract_v2.md / arbitrate_v1.md (sentinel templates)
tests/              pytest suite
data/               doc lists, facts, conflicts, graphs (parsed/ is gitignored)
doc/                thesis_proposal.tex, thesis_milestone.tex (authoritative)
PROJECT_STATE.md    detailed engineering log + design decisions
```

What is **not** committed (must be regenerated locally):

- `data/parsed/` — `ParsedDocument` JSONs from HuggingFace lex_glue. Regenerate
  with the commands in §3.1.
- `data/raw_html/`, `data/hf_cache/` — HF download caches.
- `.venv/`, `__pycache__/`.

What **is** committed for the demo path:

- `data/doc_lists/phase1_short.json` — 50-doc shortlist with metadata.
- `data/facts/qwen3.5_4b/train-000000.json` — real Qwen3.5-4B output (32 facts).
- `data/facts/{gemma3-4b-sim,phi4-mini-sim}/train-000000.json` — synthesized
  second/third annotators (see §5).
- `data/conflicts/train-000000.json` — Phase-2 output for `train-000000`.
- `data/graphs/train-000000.json` — KG snapshot for the UI.

So out of the box, after you generate the parsed JSON for `train-000000`, the
UI is fully functional in read-only mode (no Ollama required).

## 2. Prerequisites

- **Python 3.11** (3.10 should also work; tested on 3.11).
- **Git**.
- **Ollama** with the `qwen3.5:4b` model — required only if you want to actually
  re-run Phase-1 extraction or Layer-2 arbitration. The committed synthetic
  demo data lets you explore the UI without it.
- **~3 GB free disk** for the HuggingFace cache on first run.
- **GPU optional**. The whole pipeline was developed on a 6 GB RTX 3060M;
  it will also run on CPU (much slower for Phase-1).

## 3. First-time setup

```powershell
# Windows PowerShell
git clone <repo-url> MasterThesis
cd MasterThesis

python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

On Linux / macOS, replace the activation line with `source .venv/bin/activate`.

### 3.1 Generate `data/parsed/`

The UI's left text pane reads `data/parsed/<doc_id>.json`. The demo annotator
files reference `train-000000`, so you need at least that one parsed JSON
on disk. Two options:

**Option A — minimal (one doc, ~30 s):**

```powershell
python -m scripts.run_dry_run --n 5
```

Downloads the lex_glue eurlex subset on first run, parses 5 short docs into
`data/parsed/`, including `train-000000`.

**Option B — full 50-doc demo set (~2–3 min):**

```powershell
python -m scripts.select_short_docs --n 50
```

This rebuilds the same 50-doc manifest the synthetic facts were derived from
and writes all 50 parsed JSONs into `data/parsed/`. Recommended if you also
want to explore Phase-1 batch runs from the UI.

### 3.2 (Optional) Install Ollama and pull models

Only needed if you want to re-run extraction or Layer-2 LLM arbitration.

```powershell
# Install from https://ollama.com/download, then:
ollama pull qwen3.5:4b
ollama pull gemma3:4b      # optional — currently simulated
ollama pull phi4-mini      # optional — currently simulated
```

The dispatcher in `src/extractor.py` talks to `http://localhost:11434`.
Start Ollama (it auto-starts on Windows after install) before launching any
LLM-touching command.

> **Note.** Out of the box, only `qwen3.5:4b` has produced real facts; the
> gemma and phi4 annotator files in the repo are deterministic perturbations
> generated by `scripts/synthesize_facts.py`. This is a Phase-2 development
> shortcut — see §5.

## 4. Running the UI (the fastest way to see everything)

```powershell
python -m ui.app
```

Default: `http://127.0.0.1:5000/`. Flags: `--host`, `--port`, `--debug`.

Once open:

1. The doc selector starts on `train-000000`.
2. Three coupled panes: source text (left), Cytoscape KG (centre), facts
   table (right). Click any fact / mark / edge → all three panes highlight
   the same fact.
3. Top bar: annotator chips, conflict-label dropdown, merge-threshold slider
   (debounced, hot-reloads the KG), search box.
4. Bottom drawer (toggle with the chevron) has four tabs:
   - **Background runs** — kick off Phase-1 or Phase-2 batches from the UI.
   - **Guidelines** — load/edit/save `prompts/extract_*.md` versions.
   - **Run matrix** — interactive (doc × model × guideline) checkbox grid.
   - **Compare** — v1 ↔ v2 distribution-shift chart.
5. A **Live jobs** panel just under the top bar shows in-flight Phase-1 /
   Phase-2 / re-extraction jobs with progress bars.

If you only generated 1 parsed doc (Option A above), the doc selector only
shows `train-000000`. That's enough to exercise every UI feature except the
batch run matrix.

## 5. Running the pipeline from the command line

Same operations as in the UI, but headless. Useful for batch reruns or for
debugging without a browser.

### 5.1 Phase-1 — LLM fact extraction

Single (model, doc):

```powershell
python -m src.extractor extract `
    --model qwen3.5:4b `
    --doc data\parsed\train-000000.json `
    --guideline v1
```

Output: `data/facts/<model_safe>/<doc_id>.json`.

Batch driver (`models × docs`, resumable via `data/facts/manifest.json`):

```powershell
# All 3 models × the 50-doc shortlist
python -m scripts.run_extraction run

# Smoke test
python -m scripts.run_extraction run --models qwen3.5:4b --max-docs 2

# Status of the manifest
python -m scripts.run_extraction status
```

Other useful subcommands of `src.extractor`:

```powershell
python -m src.extractor ps         # which models are currently resident in Ollama
python -m src.extractor unload     # evict all to free VRAM
```

### 5.2 Synthesize multi-annotator data (development shortcut)

While only `qwen3.5:4b` has real facts, Phase-2 needs ≥ 2 annotators per doc.
The repo ships with synthetic facts for `train-000000`; to regenerate (or
extend to other docs):

```powershell
python -m scripts.synthesize_facts --doc train-000000
```

Writes `data/facts/{gemma3-4b-sim,phi4-mini-sim}/<doc>.json` (seeded, fully
deterministic). Every synthesized fact is tagged `extra.synthesized=True`.

### 5.3 Phase-2 — alignment + two-layer conflict detection

Without LLM (deterministic stub for Layer-2 — fast, fine for UI demos):

```powershell
python -m scripts.run_phase2 --doc train-000000 --skip-layer2
```

With the real Layer-2 LLM arbitrator (requires Ollama + qwen3.5:4b):

```powershell
python -m scripts.run_phase2 --doc train-000000 --layer2-model qwen3.5:4b
```

Batch over every doc with ≥ 2 annotators on disk:

```powershell
python -m scripts.run_phase2 --all --skip-layer2
```

Output: `data/conflicts/<doc_id>.json` (consumed directly by the UI).

### 5.4 Tests

```powershell
python -m pytest tests/
```

29 tests, all passing on the reference setup. They cover alignment, Layer-1
filter, KG build, coverage, and text ingestion. The SBERT-dependent paths
fall back to a sklearn TF-IDF backend when `sentence-transformers` is missing
(sandbox safety net — see `src/alignment.py`).

## 6. Suggested order of exploration

If you've only got 15 minutes and want to see the headline features:

1. Setup (§3) → generate parsed JSONs with `run_dry_run --n 5`.
2. `python -m ui.app` → open in browser.
3. Click on edges in the KG → pair-detail modal shows aligned fact pairs
   with cosine, Layer-1, and Layer-2 reasons.
4. Bottom drawer → **Guidelines** tab → open `extract_v2`, look at the diff
   vs `extract_v1` (the four target violations from `PROJECT_STATE.md` §4).
5. Bottom drawer → **Compare** tab → if you have time, run Phase-1 with both
   guideline versions from the **Background runs** tab, then re-open Compare
   to see the v1→v2 distribution shift.

If you want to reproduce the whole pipeline from scratch, the canonical
sequence is:

```powershell
python -m scripts.select_short_docs --n 50
python -m scripts.run_extraction run --models qwen3.5:4b
python -m scripts.synthesize_facts --doc train-000000          # until other models ship
python -m scripts.run_phase2 --all --skip-layer2
python -m ui.app
```

## 7. Caveats and known issues

- **EUR-Lex direct fetch is blocked.** `eur-lex.europa.eu` returns a WAF JS
  challenge to non-browser clients; CELLAR returns 400. All corpus access
  goes through HuggingFace `coastalcph/lex_glue` (text-only — annex tables
  are not in the HF version). The HTML fetcher and table linearizer are
  kept in the codebase for a future Phase-4 stress test.
- **6 GB VRAM ceiling.** Three 4 B models cannot be co-resident. The
  dispatcher loads models sequentially with `keep_alive=0` eviction.
- **Qwen3 thinking-mode policy.** Phase-1 forces `think=False` (Qwen3 returns
  empty JSON under `format="json"` otherwise). Phase-2 Layer-2 keeps thinking
  ON. Do not flip these without reading `PROJECT_STATE.md` §4.
- **Only `train-000000` has real LLM output.** Other 49 short docs are parsed
  but unannotated. Running `scripts/run_extraction run` against the full
  shortlist with all three models is on the Phase-1 to-do list.
- **`data/parsed/` is gitignored** — see §3.1.
